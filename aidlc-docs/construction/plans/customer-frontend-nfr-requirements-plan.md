# Customer Frontend - NFR Requirements 계획

## 유닛 컨텍스트
- **유닛 ID**: customer-frontend
- **기술**: React 18+, React Router, React Query, Context API, Axios, localStorage
- **기능 설계**: 완료

## 계획 개요
이 계획은 customer-frontend 유닛의 비기능 요구사항(NFR)을 평가하며, 확장성, 성능, 가용성, 보안, 기술 스택 결정에 집중합니다.

## 명확화 질문

### 확장성 요구사항

**Q1: 예상 동시 사용자 수**
레스토랑당 몇 명의 동시 고객(태블릿)이 예상되나요?
- A) 1-10개 태블릿 (소형 레스토랑)
- B) 10-50개 태블릿 (중형 레스토랑)
- C) 50-100개 태블릿 (대형 레스토랑)
- D) 100개 이상 태블릿 (초대형 레스토랑)

[Answer]: B

**Q2: 데이터 증가 패턴**
테이블 세션 동안 localStorage에 얼마나 많은 데이터가 축적될 것으로 예상되나요?
- A) 최소 (< 1MB) - 일반적인 장바구니 및 세션 데이터
- B) 보통 (1-5MB) - 장바구니 + 캐시된 메뉴 + 주문 내역
- C) 많음 (5-10MB) - 광범위한 캐싱
- D) 매우 많음 (> 10MB) - 대체 저장소 필요

[Answer]: B

**Q3: 메뉴 데이터 크기**
레스토랑당 몇 개의 메뉴 항목이 예상되나요?
- A) 소형 메뉴 (< 50개 항목)
- B) 중형 메뉴 (50-200개 항목)
- C) 대형 메뉴 (200-500개 항목)
- D) 초대형 메뉴 (> 500개 항목)

[Answer]: A

### 성능 요구사항

**Q4: 페이지 로드 시간**
고객 앱의 허용 가능한 초기 페이지 로드 시간은?
- A) < 1초 (우수)
- B) < 2초 (양호)
- C) < 3초 (허용)
- D) < 5초 (견딜 만함)

[Answer]: B

**Q5: API 응답 시간**
중요한 작업(로그인, 주문 제출)의 허용 가능한 API 응답 시간은?
- A) < 500ms (우수)
- B) < 1초 (양호)
- C) < 2초 (허용)
- D) < 3초 (견딜 만함)

[Answer]: A

**Q6: UI 반응성**
UI 상호작용(버튼 클릭, 장바구니 업데이트)의 허용 가능한 지연 시간은?
- A) < 100ms (즉각)
- B) < 300ms (빠름)
- C) < 500ms (허용)
- D) < 1초 (견딜 만함)

[Answer]: B

**Q7: 이미지 로딩**
성능 최적화를 위해 메뉴 이미지를 어떻게 로드해야 하나요?
- A) 지연 로딩 + 플레이스홀더
- B) 점진적 로딩 (저해상도 먼저)
- C) 앱 시작 시 모든 이미지 미리 로드
- D) 필요 시에만 로드

[Answer]: A

### 가용성 요구사항

**Q8: 가동 시간 기대치**
고객용 앱의 예상 가동 시간은?
- A) 99.9% (연간 8.76시간 다운타임)
- B) 99% (연간 3.65일 다운타임)
- C) 95% (연간 18.25일 다운타임)
- D) 최선 노력 (특정 SLA 없음)

[Answer]: B

**Q9: 오프라인 기능**
앱이 오프라인(인터넷 연결 없이)에서 작동해야 하나요?
- A) 예, 완전한 오프라인 모드 (메뉴 탐색, 주문 대기열)
- B) 부분 오프라인 (캐시된 메뉴만 탐색)
- C) 오프라인 모드 없음 (인터넷 연결 필요)
- D) 우아한 성능 저하 (경고와 함께 캐시된 데이터 표시)

[Answer]: B

**Q10: 세션 지속성**
브라우저 재시작 후 고객 세션이 얼마나 오래 유지되어야 하나요?
- A) 테이블 세션 종료 시까지 (관리자가 관리)
- B) 24시간
- C) 7일
- D) 무기한 (수동 로그아웃까지)

[Answer]: A

### 보안 요구사항

**Q11: 인증 보안**
어떤 수준의 인증 보안이 필요한가요?
- A) 기본 (localStorage의 JWT, HTTPS)
- B) 강화 (JWT + 리프레시 토큰, 보안 저장소)
- C) 높음 (JWT + 리프레시 토큰 + 기기 지문)
- D) 최대 (OAuth2, MFA, 하드웨어 토큰)

[Answer]: A

**Q12: 데이터 보호**
localStorage의 민감한 데이터를 암호화해야 하나요?
- A) 예, 모든 데이터 암호화 (장바구니, 세션, 토큰)
- B) 토큰만 암호화
- C) 암호화 없음 (HTTPS에 의존)
- D) 데이터 민감도에 따라 암호화

[Answer]: A

**Q13: XSS 방어**
어떤 수준의 XSS(크로스 사이트 스크립팅) 방어가 필요한가요?
- A) 기본 (React 기본 이스케이프)
- B) 강화 (CSP 헤더, 입력 살균)
- C) 높음 (CSP + DOMPurify + 엄격한 검증)
- D) 최대 (CSP + DOMPurify + WAF)

[Answer]: B

**Q14: API 보안**
어떤 API 보안 조치가 필요한가요?
- A) HTTPS + JWT 인증
- B) HTTPS + JWT + 속도 제한
- C) HTTPS + JWT + 속도 제한 + 요청 서명
- D) HTTPS + JWT + 속도 제한 + 요청 서명 + API 게이트웨이

[Answer]: C

### 기술 스택 검증

**Q15: React 버전**
React 18+가 이 프로젝트에 적합한 선택인가요?
- A) 예, React 18+ 사용 (동시성 기능, 자동 배칭)
- B) React 17 사용 (더 안정적, 적은 변경사항)
- C) 대안 고려 (Vue, Svelte, Angular)
- D) 결정하려면 더 많은 정보 필요

[Answer]: A

**Q16: 상태 관리**
React Query + Context API 조합이 상태 관리에 충분한가요?
- A) 예, 이 사용 사례에 충분
- B) 복잡한 상태를 위해 Redux 추가
- C) 더 간단한 전역 상태를 위해 Zustand 추가
- D) Context API만 사용 (React Query 없이)

[Answer]: A

**Q17: HTTP 클라이언트**
Axios가 HTTP 요청에 적합한 선택인가요?
- A) 예, Axios 사용 (인터셉터, 요청/응답 변환)
- B) 네이티브 Fetch API 사용 (더 가벼움, 내장)
- C) React Query의 내장 fetch 사용
- D) 결정하려면 더 많은 정보 필요

[Answer]: A

**Q18: 빌드 도구**
어떤 빌드 도구를 사용해야 하나요?
- A) Vite (빠름, 현대적, React에 권장)
- B) Create React App (CRA)
- C) Webpack (사용자 정의 구성)
- D) 기타 (구체적으로 명시)

[Answer]: A

### 신뢰성 요구사항

**Q19: 에러 처리 전략**
어떤 에러 처리 전략을 구현해야 하나요?
- A) 기본 (try-catch, 에러 경계, 토스트 알림)
- B) 강화 (+ 에러 로깅 서비스, 재시도 로직)
- C) 고급 (+ 에러 추적, 사용자 피드백, 분석)
- D) 최대 (+ APM, 분산 추적, 알림)

[Answer]: B

**Q20: 재시도 로직**
실패한 API 요청에 어떤 재시도 전략을 사용해야 하나요?
- A) 지수 백오프 (1초, 2초, 4초) - 최대 3회 재시도
- B) 선형 백오프 (2초, 2초, 2초) - 최대 3회 재시도
- C) 즉시 재시도 - 최대 3회 재시도
- D) 자동 재시도 없음 (수동 재시도만)

[Answer]: A

**Q21: 모니터링**
고객 앱에 어떤 수준의 모니터링이 필요한가요?
- A) 기본 (콘솔 로깅, 에러 경계)
- B) 강화 (+ Sentry 같은 에러 추적 서비스)
- C) 고급 (+ 성능 모니터링, 사용자 분석)
- D) 최대 (+ APM, RUM, 세션 재생)

[Answer]: C
