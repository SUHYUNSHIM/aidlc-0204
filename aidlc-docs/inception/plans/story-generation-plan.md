### Q1: 페르소나 세부 정보

B) **표준 페르소나** - 배경, 목표, 불만사항, 기술 수준 포함
C) **상세한 페르소나** - 인구통계, 행동 패턴, 시나리오, 인용구 포함
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: B

### Q2: User Story 형식
User Story를 어떤 형식으로 작성할까요?

A) **표준 형식** - "As a [role], I want [feature] so that [benefit]"
B) **확장 형식** - 표준 + 배경 설명 + 상세 시나리오
C) **간결 형식** - 핵심 기능 설명만 포함
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: A

---

### Q3: 수용 기준 상세도
각 User Story의 수용 기준(Acceptance Criteria)을 얼마나 상세하게 작성할까요?

A) **간단한 수용 기준** - 3-5개의 핵심 검증 항목만
B) **표준 수용 기준** - Given-When-Then 형식으로 주요 시나리오 포함
C) **상세한 수용 기준** - 모든 엣지 케이스 및 에러 시나리오 포함
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: B

---

### Q4: Story 분해 수준
큰 기능을 여러 개의 작은 스토리로 나눌까요?

A) **큰 스토리 유지** - 기능별로 하나의 큰 스토리 (예: "메뉴 관리" 하나의 스토리)
B) **중간 분해** - 주요 기능별로 분해 (예: "메뉴 등록", "메뉴 수정", "메뉴 삭제")
C) **세밀한 분해** - 모든 세부 기능을 개별 스토리로 (예: "메뉴 이미지 추가", "메뉴 가격 수정" 등)
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: B

### Q5: Story 조직 방식
User Stories를 어떻게 조직하고 그룹화할까요?

A) **페르소나 기반** - 고객 스토리, 관리자 스토리로 그룹화
B) **기능 기반** - 주문 관리, 메뉴 관리, 테이블 관리로 그룹화
C) **사용자 여정 기반** - 주문 프로세스, 운영 프로세스로 그룹화
D) **Epic 기반** - 큰 Epic 아래 관련 스토리들을 계층적으로 구성
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: C

### Q6: 우선순위 표시
MVP 범위와 향후 기능을 어떻게 구분할까요?

### Q7: 기술적 제약사항 포함
User Story에 기술적 제약사항이나 구현 힌트를 포함할까요?

A) **포함하지 않음** - 순수하게 사용자 관점만 유지
B) **최소한 포함** - 중요한 기술적 제약사항만 노트로 추가
C) **상세히 포함** - 기술 스택, API 엔드포인트, 데이터 모델 힌트 포함
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: A

### Q8: 비기능 요구사항 처리
성능, 보안 등 비기능 요구사항을 어떻게 다룰까요?

A) **별도 스토리** - 비기능 요구사항을 독립적인 스토리로 작성
B) **수용 기준에 포함** - 각 기능 스토리의 수용 기준에 비기능 요구사항 추가
C) **혼합 방식** - 중요한 것은 별도 스토리, 나머지는 수용 기준에 포함
D) **포함하지 않음** - User Stories에는 기능적 요구사항만 포함
X) 기타 (아래 [Answer]: 태그 다음에 설명해주세요)

[Answer]: C

B) **표준 수준** - 일반적인 에러 케이스 및 검증 실패 시나리오 포함
C) **포괄적** - 모든 가능한 에러, 엣지 케이스, 경계 조건 상세히 기술
